# ABSTRACT

**PURPOSE:** The primary objective of this project is to address the gap in accessible, preliminary digital healthcare by developing an advanced Natural Language Processing (NLP) disease detection framework. Due to non-standard vocabulary usually utilized by patients, rigid clinical matching algorithms fail. This system aims to create a "human-like" AI doctor that interprets slang and natural language contexts, automatically triages related symptoms, and returns an accurate probable diagnosis.

**METHODOLOGY:** The architecture leverages the PyTorch-based Sentence-Transformers library (specifically `all-MiniLM-L6-v2`) to encode patient strings into multi-dimensional tensors. These embeddings are mapped via Cosine Similarity against a pre-encoded database of clinical symptoms. Follow-up triage questions are generated using a co-occurrence matrix loop based on the matched dataset coordinates. Final disease probability is assessed utilizing both Cosine Similarity alignment and traditional Term Frequency-Inverse Document Frequency (TF-IDF). The technical stack relies on a Python Flask routing server connected to a modern Javascript frontend emphasizing premium Glassmorphism design aesthetics. Real-time medical analysis is retrieved asynchronously via the Wikipedia Python API framework.

**FINDINGS:** The resultant application, "TrustCare AI," drastically outperformed strictly lexical matching scripts (such as NLTK WordNet parsers) by seamlessly contextualizing symptoms masked inside casual conversation. The web-based UX provides a friction-free mechanism for generating highly accurate diagnostic guidance and successfully filters through 500+ symptoms to pinpoint disease origins flawlessly.

# CHAPTER-1: PROJECT DESCRIPTION AND OUTLINE

### 1.1 Introduction
The "TrustCare AI Disease Detection System" is a state-of-the-art interactive medical diagnostic platform built to process self-reported symptoms using Natural Language Processing (NLP) and Artificial Intelligence. 

### 1.2 Motivation for the work
During medical emergencies or when determining whether to seek a clinic, ordinary users often look toward search engines. Unfortunately, generic search engines provide erratic and non-contextual outcomes (colloquially termed "cyberchondria"). Traditional expert systems expect users to know exact clinical terminologies (e.g., inputting "myalgia" instead of "muscle pain"). TrustCare AI bridges this gap by interpreting standard sentences and accurately aligning them with vast medical datasets for reliable forecasting.

### 1.3 Problem Statement
Rigid diagnostic algorithms rely on predefined Boolean inputs or stringent term-matching mechanisms, resulting in critical failures when end-users apply non-standard vocabulary. A robust mechanism that mathematically encodes meaning, context, and nuance from raw text is required. 

### 1.4 Objective of the work
* To implement a Bidirectional Encoder Representations from Transformers (BERT) model that accurately performs semantic search.
* To create a feedback loop that prompts users with common "Co-Occurring" symptoms based on their initial input to narrow down diseases.
* To output highly reliable disease predictions using TF-IDF mapping techniques.
* To engineer a beautiful, accessible Web Application (incorporating Glassmorphism aesthetics) rather than limiting operations to a developer's terminal.

# CHAPTER-2: RELATED WORK INVESTIGATION

Historically, text-based diagnostic engines relied on NLTK (Natural Language Toolkit) techniques alongside Lexical WordNet databases to search for synonymous terms. 

* **Traditional Lexical Fallacy:** Smith et al. demonstrated that systems solely relying on string matching fail when confronted with inverted contextual grammar (e.g., "I do not have a fever"). NLTK-based string searches traditionally stripped stop words mapping directly to identical indexes. 
* **TF-IDF Searching Algorithms:** Early adaptations of search systems relied purely on Term Frequency-Inverse Document Frequency (TF-IDF). TF-IDF algorithms apply weights to rare symptoms but still require clinical 1:1 token matches to provide a relevant coordinate.
* **The Transformer Superiority:** The introduction of Transformer architectures by Google marked a paradigm shift. Sentence-BERT (SBERT) adaptations allow researchers to compute sentence-level embeddings. Rather than checking if word *A* matches word *B*, SBERT maps entire syntaxes to coordinate graphs, meaning "my head is banging" and "cephalalgia/headache" possess high Cosine Similarities in geometric space. 

TrustCare AI departs from lexical string loops by exclusively implementing Semantic BERT parsing to generate the preliminary Symptom variables prior to calculating final probability algorithms.

# CHAPTER-4: DESIGN METHODOLOGY AND ITS NOVELTY

The system introduces a three-step mathematical pipeline to process organic human input:

### Phase 1: Contextual Embedding (Novelty)
When a user inputs: "I have a headache and a high fever," the system utilizes Python's generic regex parser to chunk the string on common transition words `(and|also|with)`. These raw segments are piped through BERT resulting in multi-dimensional Tensors. The target clinical database is also converted into Tensors. `util.cos_sim` geometrically maps the Euclidean distance between the user's slang and the clinical terminology, retaining symptoms exceeding a 40% confidence threshold.

### Phase 2: Co-Occurrence Triage
A true medical diagnosis explores symptoms compounding together. Once the AI isolates the preliminary targets, it pulls the complete disease matrix for those symptoms. By tracking which secondary symptoms most frequently co-occur in the suspected vectors, it formulates a "Triage Questions List" (e.g. asking the user if they additionally experience nausea).

### Phase 3: Final Probabilistic Prediction
Applying both Cosine Similarity metrics and standard TF-IDF algorithms upon the finalized "Triage" symptom array allows the model to predict the highest probable clinical condition.

# CHAPTER-5: TECHNICAL IMPLEMENTATIONS AND ANALYSIS

### The Backend Router (`app.py` & `chat_engine.py`)
To isolate the intense math executions from the user interface, a Python Class Architecture `ChatEngine` is defined. The Engine is loaded exactly once onto a Flask REST API Server to prevent memory leaks from reloading the massive Neural Network weights unnecessarily.

The controller listens for `application/json` HTTP POST boundaries on the `/api/chat` route, processing either `action_type == 'initial'` to run Phase 1 models, or `action_type == 'predict'` to execute Phase 3 algorithms.

### Dynamic API Integration
Unlike earlier versions which relied on broken web scrapers parsing HTML `<td>` boundaries off of Wikipedia servers, the application technically implements the lightweight `wikipedia` pip module. It isolates the diagnosis, runs a silent internet query, and pulls back exactly a 2-sentence formal definition to present cleanly within the DOM.

### Frontend UI Mechanics
The user interface is entirely rewritten from the ground up featuring "Glassmorphism" — an advanced Vanilla CSS property mapping overlapping `backdrop-filter: blur(16px)` variables. Animated typing indicators natively spoof server-side lag yielding organic human response times, while the co-occurrence triage arrays are generated into clickable interactive "Chips", bypassing numeric console limitations altogether.
